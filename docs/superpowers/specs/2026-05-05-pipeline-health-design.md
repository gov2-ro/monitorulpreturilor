# Design Spec: Pipeline Health & Data Quality Layer

**Date:** 2026-05-05  
**Status:** Approved

---

## Context

The price monitor has accumulated 21.7M retail price records across 4,078 stores, 87K products, and 10 networks over its first few days of operation. Before surfacing any public-facing insights (cross-network comparisons, inflation tracking), we need confidence in:

1. **Price accuracy** — are fetched prices correct, or are outliers/errors slipping through?
2. **Pipeline freshness** — is every store being reached on schedule, or are some silently lagging?

This spec covers three sequenced phases: a diagnostic report (fast, no DB changes), a persistent quality layer, and public insights built on validated data.

---

## Phase 1: Pipeline Diagnostic Report

### Goal

A new script `generate_pipeline_report.py` reads the DB post-fetch and writes `docs/pipeline-health.html`. Same static-HTML pattern as `generate_site.py`. To be added to CI after the daily fetch.

### Metrics

| Check | Logic | Alert threshold |
|-------|-------|-----------------|
| **Store freshness** | Days since last price record per store | > 2 days = stale |
| **Run completion** | % of UATs/stores reached per run from `runs` table; shows interrupted runs and cutoff point | < 80% = degraded |
| **Price outliers** | Per product: flag prices > 3σ from median across all stores/dates | > 5% of records flagged = investigate |
| **Price change velocity** | % of `prices_current` records that changed vs. previous day | 0% = pipeline stuck; > 30% = bulk reprice or data error |
| **Promo depth sanity** | Promo price < 20% of that product's median regular price | Any = likely promo-flag mapping error |
| **Top anomalous products** | ~20 products ranked by outlier count | — |

### Output

Self-contained HTML with traffic-light indicators (green/amber/red), a summary table per check, and sortable detail tables. No new DB tables — purely reads existing data.

## Phase 2: Persistent Data Quality Layer

### Goal

A `price_flags` table and an updated `build_anomalies.py` (file already exists — read before extending) that persist quality flags after each fetch. All downstream analytics JOIN against this table to exclude bad records.

### New DB table

```sql
CREATE TABLE IF NOT EXISTS price_flags (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  product_id  INTEGER NOT NULL,
  store_id    INTEGER NOT NULL,
  price_date  TEXT NOT NULL,
  flag_type   TEXT NOT NULL,
  details     TEXT,  -- JSON: {"price": 0.5, "median": 12.0, "z_score": 8.2}
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(product_id, store_id, price_date, flag_type)
);
CREATE INDEX IF NOT EXISTS idx_price_flags_lookup
  ON price_flags(product_id, store_id, price_date);
```

Add to `db.py`'s `init_db()` and add an upsert helper.

### Flag types

| Flag | Trigger | Rationale |
|------|---------|-----------|
| `outlier_price` | Price > 3σ from product median across all stores | Likely fetch/parse error |
| `price_spike` | > 50% change vs same product/store previous day | Sudden spikes uncommon for groceries |
| `promo_too_deep` | Promo price < 20% of product's median regular price | Suggests promo flag misapplied |

`stale_store` is tracked at the store level in the Phase 1 report, not as a per-price-record flag.

### Clean prices query pattern

```sql
SELECT p.* FROM prices_current p
LEFT JOIN price_flags f
  ON f.product_id = p.product_id
  AND f.store_id = p.store_id
  AND f.price_date = p.price_date
WHERE f.id IS NULL;
```

### Key question this answers

PENNY shows ~70% average price spread across its stores. After Phase 2 flags are applied:
- If those wide-spread records are mostly flagged → data artifact
- If they remain clean → genuine regional pricing, a publishable finding

## Phase 3: Public Insights (built on validated data)

### Goal

New sections within existing pages (`analytics.html`, `trends.html`), generated by the existing `export_analytics.py` / `generate_site.py` pipeline. All queries exclude flagged records.

### Insights to surface

| Insight | Description | Output |
|---------|-------------|--------|
| **Network price comparison** | For each major product category, which chain is cheapest this week? Ranked table/bar chart | `analytics.html` |
| **Price change tracker** | Biggest price increases/decreases week-over-week (inflation signal) | `trends.html` |
| **Promo effectiveness** | Is the promo price at Carrefour actually lower than the regular price at Lidl? | `analytics.html` |
| **Store price index** | Rank stores by average price for a common basket of goods | `analytics.html` |

| File | Change |
|------|--------|

| `db.py` | Add `price_flags` table to `init_db()`, add upsert helper |
| `export_analytics.py` | Phase 3: filter out flagged records |
| `generate_site.py` | Phase 3: add new insight sections to HTML |
| `docs/pipeline-health.html` | Phase 1 output |

---

## Implementation order

1. Read `build_anomalies.py` to understand current state
2. Build `generate_pipeline_report.py` (Phase 1 — no DB changes, fastest value)
3. Run Phase 1, review findings, calibrate thresholds for Phase 2
4. Add `price_flags` to `db.py`, extend `build_anomalies.py` (Phase 2)
5. Run Phase 2, inspect flag counts, answer the PENNY question
6. Extend analytics generation with clean-data filters (Phase 3)
