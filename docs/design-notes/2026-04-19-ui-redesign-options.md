# UI redesign — options explored (2026-04-19)

Context: the user asked for a fresh UI with the ethos of datawrapper.de, FiveThirtyEight (old),
and pudding.cool. Current app: 18 static HTML pages generated by `generate_site.py` from SQLite,
served on GitHub Pages. Vanilla HTML/CSS/JS + Chart.js + Leaflet via CDN. No bundler.

This document captures the options considered before deciding, so we can revisit them later
without re-deriving the trade-offs.

## Decision (2026-04-19)

- **Stack: A — Keep static (Python → HTML, hand-crafted CSS).**
- **Positioning: Editorial-led homepage, with a separate `/tablou` dashboard for power users.**

Reasons: matches the cited references' actual stacks (handcrafted CSS + D3/Chart.js is the
journalism-web default); preserves the civic / "view source is legible" ethos; keeps the
daily CI pipeline simple; ceiling on look is high; editorial voice is distinctive where
a dashboard is generic.

## Stack options

### A — Keep static, hand-crafted CSS  *(chosen)*

What changes visibly: almost nothing is out of reach. Datawrapper, old-538, Pudding,
ProPublica, NYT interactives all run on "hand-crafted HTML/CSS + D3/Chart.js + a little
vanilla JS." A strong design system (typography scale, color tokens, spacing, chart
wrappers) buys the entire ceiling.

Hard things:

- Rich client-side state (a product search across 6,932 products with URL params +
  localStorage). Doable in ~300 lines of vanilla JS, but tedious.
- Authoring new narrative stories means hand-writing HTML (no Markdown with inline charts).
- No component reuse across pages — reuse lives in Python template helpers.

Wins: zero npm, zero build rot, single Python CI job, legible source, low long-term
maintenance cost.

Influence on look: near zero. The ceiling is design skill, not the stack.

### B — Tailwind + Alpine via CDN  *(rejected)*

Buys: faster iteration on generic components (menus, modals, tooltips).

Costs: ~300KB Tailwind CDN (vs ~15KB handcrafted CSS) hurts on Romanian 4G; `class="flex
gap-4 text-slate-700 font-semibold tracking-tight"` soup kills the civic-transparency
signal; Tailwind's defaults (`rounded-md shadow-sm`, slate palette) are instantly
recognizable as "AI/bootstrap era" and fight a Datawrapper-ish look. Net-net: MORE CSS
than starting from zero with a design system.

Verdict: small benefit for small features, loses the ethos. The "move sideways" option.

### C — Astro / 11ty (modern build)  *(deferred)*

Buys: real component reuse (`<StatCard>`, `<NetworkBadge>` shared across pages); **MDX
for stories** (Markdown with embedded interactive chart components — Pudding's actual
workflow); TypeScript, linting, hot reload; partial hydration.

Costs: full rewrite of the generator; Node/npm on the critical path; harder to maintain
for a Python-first author.

Influence on look: only matters for **interactive stories**. For dashboards/tools the
output is identical to A.

Revisit when: you want to regularly publish hand-authored, scroll-driven Pudding-style
pieces (a sustained content commitment, not a one-off).

## Positioning options

### Editorial-led briefing  *(chosen)*

Homepage = "Buletinul prețurilor". Hero: headline + lede + one annotated chart. Below:
4 stat tiles, 3 story cards, tool grid, compact dashboard strip, footer.

Consumer (Ioana, shopping): reads the hero, clicks into a tool from the "Instrumente"
section.

Journalist (Andrei): the hero IS the product — reads, downloads data, shares URL.

Repeat user (Claudia): scrolls past hero, clicks tool from top nav.

Pros: strong voice + identity; social-shareable daily; builds civic credibility (stories +
methodology + open data linked from hero). Mobile-friendly (one headline + chart + stats).

Cons: auto-generated lead has to be consistently good. Mitigation: rank existing 5 auto-
stories daily, promote the best as the lead, keep others as cards. Consumer tools are one
extra tap (mitigated by persistent top nav).

### Tool-led dashboard  *(rejected for homepage, preserved as `/tablou.html`)*

Homepage = KPI tiles + price index + fuel trends + coverage bars. Like current
`index.html` but better designed.

Pros: every number visible immediately; expected pattern; simpler to build.

Cons: generic (100 government dashboards look like this); no voice; nothing shareable
except "look at this dashboard"; abandons the references' ethos.

Revisit when: user research shows the homepage's primary traffic is power users who come
back daily for numbers (not readers / new visitors).

### Hybrid split  *(rejected)*

Top: lead story + chart. Below fold: KPI dashboard + tool grid.

Pros: both personas in one scroll.

Cons: longer, busier page; no single focal moment; phone viewport becomes a scroll
marathon; compromise solutions often feel compromised.

Revisit when: analytics show most homepage visitors scroll past the hero AND engage with
a dashboard below (i.e., they tolerate the length).

## Reopen triggers

Reopen this decision if **any** of:

- Auto-generated stories are too weak to consistently anchor the homepage (→ Hybrid).
- Consumer analytics show homepage traffic bounces at the hero because they want tools
  directly (→ consider Tool-led or Hybrid).
- A content commitment emerges for weekly hand-authored interactive stories (→ Astro/C).
- Mobile performance suffers from font + CSS weight beyond acceptable budgets (→ stricter
  Stack A discipline, or reconsider C with per-route hydration).
